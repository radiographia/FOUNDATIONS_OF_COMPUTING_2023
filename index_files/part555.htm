Implementation of Access Matrix as Access Control Lists‌

In practice, an access matrix is large and sparse. Most domains have no access at all to most objects, that is, most entries of an access matrix are empty. Therefore, direct implementation of an access matrix as a two-dimensional matrix would be inefficient and expensive (wastage of disk space). Access control lists (ACLs) is a method used for implementing an access matrix. This method decomposes the access matrix by columns, and implements each column of the matrix as an access list for the object corresponding to that column. It does not store empty entries of the matrix in the access list. Therefore, for each object, the system maintains a list of ordered pairs (domain, rights), which defines all domains with a nonempty set of access rights for that object. Further details of working and properties of a security system based on ACLs are as follows.

1. Access validation. Whenever a subject in domain D executes an operation r on an object O, the system first searches the access list for object O for the list element whose domain field is D. It then searches the rights field of this element for r. If found, it allows the operation to continue; otherwise, it returns a protection violation message to the subject.

In this method, the system checks the access list on every access. This is a desirable feature from security point of view. However, consulting the access list on every access could cause substantial overhead, especially when the access list is long. The method overcomes this drawback by maintaining a cache for the access list entries of only active domains.

2. Granting rights. The system grants access right r for object O to domain D in the following manner:

a. It first searches the access list for object O for the list element whose domain field is D.

b. If found, it adds right r to the rights field of this list element. Otherwise, it adds a new list element to the access list for object O, and sets domain field and a rights field of this list element to D and r, respectively.

3. Passing rights. The system passes (propagates) access right r for object O from a domain

D1 to another domain D2 in the following manner:

a. It first checks the access list for object O to ensure that D1 possesses either owner

right for object O or copy right for access right r.

b. If D1 possesses any of the above two rights, it grants access right r for object O to domain D2 in the above-described manner. Otherwise, it returns a protection violation message.

4. Rights revocation. The system revokes access rightr for object O from domain D by deleting r from the rights set of domain D in the access list for O.

For file protection, several systems use user-oriented domains like that of UNIX in which a (uid, gid) pair forms a domain. Revocation of a user's access right becomes more complicated in these systems because access revocation requires deletion of the user's uid (if present) from the access list of the object in question and cancellation of the user's membership from all groups that belong to a domain that has access to that object. In a large system, the process of discovering all groups from which to remove the user and performing the removal operation may take a significant amount of time that may be unacceptable in emergencies. Some systems use the concept of negative rights to overcome this problem. The idea behind this concept is that to revoke a user's access right to an object, the system can give negative rights on that object to the user. Negative rights indicate denial of the specified rights, with denial overriding possession in case of conflict. With this extension, ACLs may contain negative rights, so that it is possible to express facts, such as every user of a domain except user-i and user-j may exercise right r on object O. Union of all negative rights specified for a user subtracted from his/her positive rights gives his/her actual total rights. Negative rights thus act as a mechanism for rapid and selective revocation and are particularly valuable in a large system.
