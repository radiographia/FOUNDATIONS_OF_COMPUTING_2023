Access Matrix‌

In domain-based protection approach, for each domain, a system must keep track of which rights on which objects belong to that domain. Access matrix model represents this information as a matrix, called access matrix, which has the following form:

1. The rows of access matrix represent domains.

2. The columns of access matrix represent objects.

3. Each entry in access matrix consists of a set of access rights.

4. The (i, j)th entry of access matrix defines the set of operations that a process, executing in domain Di, can perform on object Oj.

At any instance of time, the contents of the access matrix define the protection state of the system. The access matrix of Figure 15.8 shows the protection state of the system in Figure 15.7.

When a system uses an access matrix to represent its protection state, it must resolve the following issues:

1. How to decide the contents of access matrix entries?

2. How to validate access to objects by subjects?

3. How to allow the subjects to switch domains in a controlled manner?

4. How to allow changes to the protection state of the system in a controlled manner? Some normally used methods to handle these issues are described here.


Figure 15.8. Access matrix for the protection state of the system in Figure 15.7.


Deciding the Contents of Access Matrix Entries

Policy decisions concerning which rights should be part of an entry (say (i, j)th entry) of an access matrix are system dependent. However, in general, users decide the contents of entries corresponding to user-defined objects, whereas the system decides the contents of entries corresponding to system-defined objects. For example, when a user creates a new object Oj, column j is added to the access matrix with suitable entries as decided by the user's specification of access control for the object.


Validating Access to Objects by Subjects

Given a system's access matrix, how does the system allow access to objects by subjects only in the manner permitted by the protection state defined by the matrix? For this, the system associates an object monitor with each object type (see Figure 15.9), and validates every attempted access by a subject to an object in the following manner:

1. A subject S in domain D initiates r access to object O, where r belongs to the set of operations possible on O.

2. Protection system forms the triple (D, r, O) and passes it to the object monitor of O.

3. Object monitor of O looks for operation r in (D, O)th entry of access matrix. If present, the system permits the requested access; otherwise, it returns a protection violation message to S.


Figure 15.9. Validation of access to objects of an object type by subjects.


Allowing Controlled Domain Switching

The principle of least privilege requires that a protection system should allow a process to switch from one domain to another during its lifetime so that at any time the process has only as much rights as is necessary for performing its task at that time. However, the system must properly control domain switching by processes; otherwise, a process may switch to a powerful domain and violate protection policies of the system.

A protection system controls domain switching by treating domains as objects on which the only possible operation is switch. Hence, it also includes domains as objects in access matrix.

Figure 15.10 shows the access matrix of Figure 15.8 with the three domains also as objects. The system permits domain switching from domain Di to domain Dj, if and only if the right switch is present in (Di, Dj)th entry of its access matrix. Hence, in the example system of Figure 15.10, a process executing in domainD1 can switch to domain D2 orD3, a process executing in domain D2 can switch to domain D3 only, and a process executing in domain D3 cannot switch to any other

domain.


Figure 15.10. Access matrix of Figure 15.8 with domains included as objects.


Allowing Controlled Change to Protection State

In a flexible design, a protection system should also allow changes to the content of a domain. However, this facility is not essential because if a process cannot change the content of a domain, it can provide the same effect by creating a new domain with the changed contents, and switching to that domain when it wants to change the domain contents. Although not essential, a protection system normally provides this facility for greater flexibility. To provide this facility, it treats the access matrix itself as an object, which needs protection. In fact, since each entry in access matrix is modifiable individually, it treats each entry as an object, which needs protection. For allowing controlled change, it defines copy, owner, and control as possible rights on this new object. To simplify the description, let us categorize allowing changes to the content of access matrix entries into two types - allowing changes to column entries and allowing changes to row entries.

1. Allowing changes to column entries. Copy and owner rights allow a process to change entries in a column. The ability to copy an access right from one domain (row) to another is denoted by appending an asterisk (*) to the access right. For example, Figure 15.11 shows the access rights of only the first three objects (F1, F2, and F3) of the access matrix

of Figure 15.8 with some copy rights. A process executing in domain D1 can copy Read and Write operations on F1 to any other domain (any other entry in column F1), whereas a process executing in domain D2 can copyRead operation on F1 and Read and Execute operations on F2 to any other domain. The system does not allow copying of any of the operations on F3 to any other domain.

Copy right may have the following three variants:

a. Transfer. In this case, when a right is copied from (i, j)th entry to (k, j)th entry of an access matrix, it is removed from the (i, j)th entry.

b. Copy with propagation not allowed. In this case, when right R* is copied from (i, j)th entry to(k, j)th entry, only the right R (not R*) is created in (k, j)th entry. The implication of this is that a process executing in domain Dk cannot copy right R further.

c. Copy with propagation allowed. In this case, when rightR* is copied from (i, j)th entry to(k, j)th entry, right R* is created in (k, j)th entry, so that a process executing in domain Dk can copy right R* or R further.

On the other hand, owner right allows adding/deleting of rights to column entries in a controlled manner. If owner right is in (i, j)th entry of access matrix, a process executing in domain Di can add and delete any right in any entry in column j. Figure 15.12 shows the access matrix of Figure 15.11 with owner rights included for the three objects. In the figure, domains D1, D2, and D3 haveowner rights for objects F1, F2, and F3, respectively. Therefore, a process executing in domain D1 can add and delete any valid right of F1 in

any entry in column F1. Similarly, a process executing in domain D2 can add and delete any valid right of F2 in any entry in columnF2, and a process executing in domain D3 can add and delete any valid right of F3 in any entry in column F3.


Figure 15.11. An access matrix with copy rights.


Figure 15.12. An access matrix with owner rights.

2. Allowing changes to row entries. Control right, which is applicable only to domain objects, allows a process to change entries in a row. If control right is in (Di, Dj)th entry of an access matrix, a process executing in domain Di can remove any access right from row Dj. For example, Figure 15.13 shows the access matrix of Figure 15.10 with control rights. In this figure, since entries (D1, D2) and (D1, D3) have control rights, a process executing in domain D1 can delete any right from rows D2 and D3. Similarly, since entry

(D2, D3) has control right, a process executing in domain D2 can delete any right from row

D3.
