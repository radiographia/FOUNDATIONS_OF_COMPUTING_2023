Web Services‌

Web-based systems were originally designed to present information to Web surfers on the Internet. Since the early 1990s, Web technologies to build Web-based systems has gone through several generations of evolutions that range from static HTML (Hyper Text Markup Language) pages to Web-enabled business components that are at the foundation of currently popular Web Services (WS).

Early Web-based systems had a typical client-server configuration (also called two-tier model) in which a Web client had a Web browser to enable users to visit Web sites (Web servers) residing on different computers [see Figure 13.2(a)]. A Web server had HTTP (Hyper Text Transfer Protocol) server, CGI (Common Gateway Interface) programs, and shared information stored as files or database. The Web browser enabled a Web client to send HTML (Hyper Text Markup Language) forms to a Web server for processing by CGI programs. In response, the Web server sent HTML files containing JavaScripts (small pieces of code) to the Web client, which interpreted the JavaScript to present the desired information to a user.

The two-tier model did not support the high quality requirements of modern Web-based applications. It failed on security, scalability, maintainability, reliability, and availability. Hence, designers of Web-based systems first expanded the two-tier model to a three-tier model, and now more generally to an n-tier model. In the expanded model, most of the software on a Web server has been moved to a separate computer called application server and the resources (information) are also moved to different computers called resource servers (such as a database server or a file server). The new middleware software handles communication, data translation, and process distribution among the Web, application, and resource servers [see Figure 13.2(b)]. This configuration has better security, reliability, availability, maintainability, and scalability features because of the ability to separate presentation (on Web server), business logic (on application server), and data (on resource server).

Parallely, Web-based systems started using distributed object technologies for development of more sophisticated Web-enabled applications. Distributed objects are objects dispersed across a distributed system. A user having access rights for accessing an object can access it from any node of the distributed system. An object on one node can send messages to objects on other nodes. Thus, the entire distributed system appears to its users as a collection of objects. Support of distributed object-based applications requires special-purpose middleware that allows remote objects to locate, bind, and communicate with each other. A common mechanism used by such middleware is an Object Request Broker (ORB) that is responsible for these operations. Examples of middleware for distributed objects include Common Object Request Broker Architecture (CORBA) from the Object Management Group (OMG), Distributed Computing Object Model (DCOM) from Microsoft, and Remote Method Invocation (RMI) from Sun Microsystems.

While distributed object-based systems provided a basic framework for building enterprise-wide distributed applications, they did not succeed in implementing business-to-business (B2B) or global applications because distributed object-based technologies were not integrated well with the Internet. For example, CORBA had its proprietary protocol, called Internet Interoperable Protocol (IIOP), for communications between CORBA clients and servers. Hence, implementation of ORBs across organizational boundaries was difficult. Designers later introduced the concept of components (also called business objects) to address this issue. A component is a self-contained piece of software that has a well-defined interface and can be developed, delivered, installed, and run independently.

Business objects can be Web enabled to provide highly reusable business services to Web users. This idea later became popular as Web Services (WS). WS is a result of convergence of the Web and distributed object technologies. Designers of modern Web-based systems use it for building large Web-based applications for e-commerce, information distribution, entertainment, collaborative working, surveys, and numerous other activities. It allows anyone to create an electronic service and anyone else to use it, thus enabling anyone-to-anyone communications, coordination, and interaction.
