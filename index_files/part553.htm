Protection Domains‌

The principle of least privilege requires that at any time a subject should be able to access only those objects that it requires currently to complete its task. Therefore, from time-to-time, subjects may need to change the set of access rights they have to objects, depending on the task that they have to do at any time. Researchers introduced the concept of domain to provide this type of flexibility of access control in a security system.

A domain is an abstract definition of a set of access rights. That is, a set of (object, rights) pairs defines a domain. Each pair specifies an object and one or more operations that a subject operating in that domain can perform on the object. Each one of the allowed operations is called a right.

A domain-based security system defines a set of domains with each domain having its own set of (object, rights) pairs. At any time, each process in the system executes in one of the protection domains, and access rights of a process are equal to the access rights defined in the domain in which it is at that time. A process can also switch from one domain to another during execution.

Figure 15.7 shows an example of a system having three protection domains D1, D2, D3. Note the following points from this example:

1. Domains need not be disjoint. The same access rights may exist in two or more domains simultaneously. For instance, access right(Semaphore-1, {Up, Down}) is in both domains D1 and D2. This implies that a subject in either of the two domains can perform Up and

Down operations on Semaphore-1.

2. The same object can exist in multiple domains simultaneously with different rights in each domain. For instance, rights forFile-1 and File-2 are different in D1 and D2, and rights forTapeDrive-1 are different in D2 andD3. Considering File-1, a subject in domainD1 can

Read, Write, and Execute it, but a subject in domain D2 can perform only Read and Write

operations on it. Therefore, in order to execute File-1, a subject must be in domain D1.

3. If an object exists in a single domain, only the subjects in that domain can access it. For instance, only the subjects in domain D3 can access File-3.

Since a domain is an abstract concept, its realization, and rules for domain switching are system dependent. For instance, some ways of realizing a domain may be:

1. Each user as a domain. In this case, the system assigns a process to a domain based on the identity of the user on whose behalf the process is executing. A domain switching occurs when a user logs out and another user logs in.

2. Each process as a domain. In this case, access rights of a process are limited to the access rights of its own domain. A domain switching occurs when one process sends a message to another process and then waits for a response.

3. Each procedure as a domain. In this case, each procedure has its own set of access rights and a domain switching occurs when a procedure calls another procedure during the course of its execution.

UNIX uses both the concepts of user-oriented domains and procedure-oriented domains. Let us first see the use of user-oriented domains. UNIX associates a user-id (uid) and a group-id (gid) with each user. A (uid, gid) pair identifies the list of objects and types of operations that a process can perform on these objects. The domain of a process depends on the user on whose behalf it is executing. Therefore, a process's (uid, gid) pair defines its domain. That is, two processes with the same (uid, gid) pair will have the same access rights, and two processes with

different (uid, gid) pairs will have different access rights. In the latter case, several access rights may be the same for the two processes. A process switches domains by acquiring a new uid or gid by executing thesetuid or setgid commands. Furthermore, note that superuser in a UNIX system is not a person but a name for a special domain within which the most privileged processes, needing access to all objects in the system, may run.

Now let us see the use of procedure-oriented domains. UNIX groups all procedures in two classes - user procedures and kernel procedures. The two classes form two domains called user mode and kernel mode. A process running in kernel mode has a different set of access rights as compared to a process running in user mode. For example, in kernel mode, a process can access all pages in physical memory, on entire disk area, and all other protected resources. In this two- domain architecture, when a user process does a system call, it switches from user mode to kernel mode. Therefore, at any time, the domain of a process (and hence its access rights) depends on whether it is executing a user procedure or a kernel procedure at that time.
