IPSec Functionality‌‌

IPSec provides the following three functionalities:

1. Authentication. It enables the receiver of a packet to get assured that the packet was indeed sent by the party (sender) identified as the source in the packet header.

2. Confidentiality. It enables communicating entities to encrypt messages to prevent eavesdropping by third parties.

3. Key management. It enables secure exchange of keys between communicating entities to establish a secure communication channel between them.

IPSec establishes a Security Association (SA) between two entities that want to transfer data between them securely. An SA is a one-way connection (simplex connection) between two entities and has a security identifier associated with it. Hence, an SA allows data transfer in one direction only (sender to receiver). If two-way secure exchange of data is required (peer relationship), then two SAs are required, one for each direction.

IPSec has the following two main parts:

1. The first part is in the form of two new headers, which contain information about security identifier, integrity control data, and other information. Each packet has these two headers.

2. The second part is called ISAKMP (Internet Security Association and Key Management Protocol). It deals with establishing matching keys between two communicating entities to create a secure communication channel between them.

IPSec supports the following two usage modes:

1. Transport mode. In this mode, IPSec inserts the IPSec header just after the IP header, and modifies the “Protocol” field in the IP header to indicate that there is an IPSec header after the normal IP header. The IPSec header contains security information such as the SA identifier, a new sequence number, an integrity check of the payload (message content), etc. Typically, distributed systems use transport mode for end-to-end communication between two hosts (for example, a client and a server, or two workstations).

2. Tunnel mode. In this mode, IPSec encapsulates the entire IP packet in the body of a new IP packet with a completely new IP header. The entire original (or inner) packet travels through a “tunnel” from one point of an IP network to another (no routers along the way are able to examine the inner IP header). In this way, it provides protection to the entire IP packet. Moreover, since the original packet is encapsulated, the new (larger) packet may have different source and destination addresses from that of the original packet. This further adds to the security.

Tunnel mode is useful when one or both ends of an SA are a security gateway (such as a firewall or router) that implements IPSec. In this mode, the gateway encapsulates and decapsulates packets as they pass through it. Hence, a number of hosts on the networks behind firewalls may engage in secure communications without implementing IPSec. The

unprotected packets generated by such hosts travel safely through external networks by tunnel mode SAs setup by the IPSec software in the firewall or secure router at the boundary of the local network.

Tunnel mode is also useful when a communication system aggregates several TCP connections and handles them as one encrypted stream. This is often done to prevent intruders from doing traffic analysis (study flow patterns of packets between different sources and destinations in a network) to draw certain inferences.

The main disadvantage of tunnel mode is that it adds an extra IP header, thus increasing packet size substantially. Transport mode does not affect packet size as much.
